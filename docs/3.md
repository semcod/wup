# stworz plan refaktoryzacji zakladajac, ze bedziemy generowc plik wup.yaml w ktorym bedzie mozliwa zmiana workflow sposobu dzialania wup, scope, , foldery, wykluczenia itd

Świetny pomysł – `wup.yaml` jako centralny plik konfiguracyjny pozwoli Ci **deklaratywnie** kontrolować:

- scope‑y usług,
- foldery do watchowania / wykluczenia,
- strategię testów (quick / detail / CI / offline),
- integrację z TestQL.

Poniżej konkretny **plan refaktoryzacji po krokach**, z zamysłem, że po jego wykonaniu masz:

- `wup.yaml` z sekcją `services`, `paths`, `excludes`, `test_strategy`
- `wup` czytający i walidujący ten plik,
- `TestQLWatcher` reagujący na te reguły.

***

## 1. Projekt: nowa struktura i plik `wup.yaml`

Ustalmy schemat early‑design:

```yaml
# .wup.yaml (lub wup.yaml)
project:
  name: "oqlos/testql"
  description: "Legacy monorepo with 200+ endpoints"

watch:
  # foldery obserwowane
  paths:
    - "app/**"
    - "routes/**"
    - "schema/**"
    - "!tests/**"         # wykluczenie
    - "!node_modules/**"

  # filtry globalne
  exclude_patterns:
    - "*.md"
    - "*.txt"
    - "migrations/**"

services:
  # usługi z regułami
  - name: "users"
    root: "app/users"
    paths:
      - "app/users/**"
      - "routes/users/**"
      - "schema/users/**"
    quick_tests:
      scope: "read,write"
      max_endpoints: 3
    detail_tests:
      scope: "all"
      max_endpoints: 10
    cpu_throttle: 0.7
    notify:
      type: "http+file"
      url: "http://localhost:8001/notify"
      file: "wup/notify-users.json"

  - name: "auth"
    root: "app/auth"
    quick_tests:
      scope: "auth"
    detail_tests:
      scope: "auth,read"

test_strategy:
  quick:
    debounce_s: 2
    max_queue: 5
    timeout_s: 10
  detail:
    debounce_s: 10
    max_queue: 1
    timeout_s: 30

testql:
  # ogólne opcje dla TestQL
  scenario_dir: "scenarios/tests"
  smoke_scenario: "smoke.testql.toon.yaml"
  output_format: "json"
  extra_args:
    - "--timeout 10s"
```

Założenia:

- `paths` i `exclude_patterns` używają globów (`**`).
- każda usługa ma `quick_tests` i `detail_tests` (scope + limity).
- `test_strategy` to globalne ustawienia dla warstw.

## 2. Plan refaktoryzacji (krok po kroku)

### Krok 0: Format i ładowanie konfiguracji

1. **Dodaj miejsce na konfigurację**
    - Utwórz `wup/config.py`:
        - `load_config(project_root: Path) -> dict` – szuka `wup.yaml` / `.wup.yaml` i ładuje `yaml.safe_load()`.
        - `validate_config(raw: dict) -> WupConfig` – dataclass / pydantic‑like struktura (pola `project`, `watch`, `services`, `test_strategy`, `testql`).
2. **Przykładowy model** (w pseudokodzie):

```python
# wup/models/config.py
class WatchConfig:
    paths: list[str]
    exclude_patterns: list[str]

class ServiceConfig:
    name: str
    root: str
    paths: list[str]
    quick_tests: dict
    detail_tests: dict
    cpu_throttle: float
    notify: dict

class TestStrategyConfig:
    quick: dict
    detail: dict

class TestQLConfig:
    scenario_dir: str
    smoke_scenario: str
    output_format: str
    extra_args: list[str]

class WupConfig:
    project: dict
    watch: WatchConfig
    services: list[ServiceConfig]
    test_strategy: TestStrategyConfig
    testql: TestQLConfig
```

3. **Załaduj w `core.py`**
    - `def init_watcher(project_root: Path) -> WupWatcher`:
        - ładuje `wup.yaml` → `WupConfig`
        - inicjalizuje `TestQLWatcher(config.testql, config.test_strategy, config.services)`

***

### Krok 1: Refaktorowanie `core.py` (watcher + queue)

1. **Zamiast zakodowanych ścieżek**
    - wytnij twarde `paths = ["app/commands", "app/discovery"]`
    - zrób:

```python
def build_watched_paths(config: WupConfig) -> list[str]:
    all_paths = list(config.watch.paths)
    # w pasma z `services[*].paths`, które są w `watch.paths`
    for svc in config.services:
        for svc_path in svc.paths:
            if svc_path in self.config.watch.paths:
                all_paths.append(svc_path)
    return all_paths
```

2. **Mapowanie pliku → usługa**
    - dodaj `def infer_service(path: Path, config: WupConfig) -> ServiceConfig | None`:
        - sprawdza `p.path` względem `svc.root` / `svc.paths` i zwraca konkretną usługę.
3. **Przepisz `test_queue` wg konfiguracji**:
    - dla `quick_test` korzystaj z `svc.quick_tests.scope` i `max_endpoints`
    - dla `detail_test` używaj `svc.detail_tests.scope` i `max_endpoints`

***

### Krok 2: Integracja `wup.yaml` z `TestQLWatcher`

1. **Zmodyfikuj `TestQLWatcher`**
    - konstruktor przyjmuje `WupConfig` i `TestQLConfig`:

```python
class TestQLWatcher:
    def __init__(self, config: WupConfig, testql_config: TestQLConfig):
        self.config = config
        self.testql_config = testql_config
        self.test_queue = deque()
```

2. **Pobieranie testów z TestQL**
    - użyj `testql` z `testql_config`:

```python
def get_endpoints_for_service(self, service_name: str) -> list[dict]:
    cmd = [
        "testql", "endpoints", f"{self.testql_config.scenario_dir}",
        "--service", service_name,
        "--scope", "...",
    ]
    # exec + parse
```

3. **Dynamiczne scope’y**
    - w `TestQLWatcher.schedule_quick_test(service)`:
        - przekaż `service.quick_tests.scope` jako `SCOPE_FILTER` do TestQL.

***

### Krok 3: Dodanie `wup map-deps` opartego na `wup.yaml`

1. **Nowa komenda**
    - `wup map-deps project_root`:
        - ładuje `wup.yaml`
        - dla każdej usługi:
            - wywołuje `testql endpoints <path> --framework ...`
            - mapuje `endpoint` → `service`
        - zapisuje `wup-deps.json` z dạngią:

```json
{
  "services": {
    "users": {
      "endpoints": ["/api/users", "/api/users/{id}"],
      "scope": {"read", "write", "auth"}
    }
  }
}
```

2. **Zalety**:
    - `map-deps` respektuje `paths` i `services` z `wup.yaml`.
    - przyszłe zmiany w konfiguracji nie wymagają update’u kodu, tylko `wup.yaml`.

***

### Krok 4: Scope‑aware TestQL + CLI Filtry

1. **TestQL‑DSL**
    - rozszerz `smoke.testql.toon.yaml` o `tags` (scope) i `CONFIG[scope]`:

```testtoon
CONFIG[^1]{key, value}:
  scope, ${SCOPE_FILTER}
  max_endpoints, ${MAX_ENDPOINTS}

API[?]{method, endpoint, tags}:
  GET, /api/users,      read, read
  POST, /api/users,     write, write
```

2. **Komenda**

```bash
testql scenarios/tests/smoke.testql.toon.yaml \
     --scope=users:read \
     --max-endpoints=3
```

3. **`wup` używa tego**:
    - `TestQLWatcher` przekazuje `scope` z `svc.quick_tests.scope`.

***

### Krok 5: Regresje i powiadomienia oparte na `wup.yaml`

1. **`notify` w każdej usłudze**
    - w `wup.yaml`:

```yaml
notify:
  type: "http+file"
  url: "http://localhost:8001/notify"
  file: "wup/notify.json"
```

2. **`RegressionsLog`**
    - `wup/regressions.py`:
        - `write(regression: dict, notify: NotifyConfig)`
            - jeśli `type` zawiera `http` → HTTP POST na `url`
            - jeśli `type` zawiera `file` → zapisuje do `file`
    - `TestQLWatcher` wywołuje `RegressionsLog.write(...)` po `detail test` failu.

***

### Krok 6: Dashboard / CLI oparte na `wup.yaml`

1. **`wup dashboard`**
    - czyta `wup.yaml`:
        - listuje wszystkie `services`
        - używa `watch.paths` do pokazania “co jest watchowane”
    - renderuje w terminalu:

```text
🖥️  WUP Dashboard
Project: oqlos/testql
Services:
  users:  ✅ OK (3/3)
  auth:   ⚠️ 1 failed
Watch paths:
  app/**, routes/**, schema/**
Excludes:
  *.md, *.txt, migrations/**
```

2. **`wup --config`**
    - wypisuje `wup.yaml` z podglądem mapy usług ↔ endpointów.

## 4. Co dalej (drugie iteration)

Po tym planie możesz kolejno:

- dodać `wup.yaml`→ IDE integracje (pyright / LSP, żeby podświetlać usługi),
- stworzyć `wup --validate` (sprawdza konfigurację i mappę `testql` ↔ `wup.yaml`),
- wdrożyć `wup ci-run` w CI, który korzysta z tych samych reguł Service ↔ TestQL.

Jeśli chcesz, mogę teraz spisać **konkretny patch** `wup.yaml` + `wup/config.py + core.py`, żebyś miał gotowy snippet do wklejenia.
