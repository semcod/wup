- **Inteligencja**: Automatyczne wykrywanie usług z ścieżek plików

## 1. Bardziej inteligentne strefowanie zmian

Obecnie WUP + TestQL traktuje **ścieżki plików → usługa → endpoints**. Możesz to wzbogacić:

- **Klasyfikacja plików**:
    - `model/*`, `orm/*` → testy tylko endpointów MUTUJĄCYCH (`POST/PUT/DELETE`)
    - `routes/*`, `views/*` → testy wszystkich endpointów tej usługi
    - `schema/*`, `dto/*` → testy walidacji payloadu dla wszystkich endpointów tej usługi
    - `tests/*` → update tylko cache’u TestQL, nie wywołuj testów
- **Typ zmiany** (heurystyka z `git` / `diff`):
    - jeśli zmiana dotyczy tylko dokumentacji / komentarzy → tylko quick test, bez detail
    - jeśli dotyka `schema` / `validators` → wymuś detail test, nawet jeśli quick test przeszedł

Możesz np. dodać w `TestQLWatcher`:

```python
self.file_type_map = {
    "orm": "model/*",
    "schema": "schema/*",
    "route": "routes/*",
    "view": "views/*",
    "test": "tests/*",
}
```

i w `infer_service` dodatkowo zwracać `file_type`, który wpływa na `test_type` w kolejce.

## 2. Historia regresji i “blame-like” tracking testów

Skoro quick test wykrywa awarię, a detail test jest już uruchamiany, możesz utrzymywać prosty **owież”**:

- **plik `regressions.json`**:

```json
{
  "2026-04-29T16:00:00": {
    "file": "app/users/routes.py",
    "service": "users",
    "failed_endpoints": ["/api/users", "/api/users/{id}"],
    "last_passing_commit": "a1b2c3d",
    "last_broken_commit": "e4f5g6h"
  }
}
```

- **Komenda**:

```bash
wup regressions --show last --service users
wup regressions --blame /api/users/{id}
```

- **Integracja z TestQL**:
    - `testql ... --output json` → parsuj `status`, `failed_assertions` i dopisz do `regressions.json`
    - w `TestQLWatcher` dodaj `self.reg_log = RegressionsLog("regressions.json")`

To daje szybki “who broke what” bez CI, tylko w lokalnym dev‑loopie.

## 3. Granulacja testów: TestQL + “scopes”

Możesz zdefiniować w `smoke.testql.toon.yaml` **logiczne scope’y**:

```testtoon
# SCENARIO: Scoped API Smoke
# TYPE: api
CONFIG[^3]{key, value}:
  scope, ${SCOPE_FILTER}
  depth, quick

API[?]{method, endpoint, tags}:
  # endpoint tags: "read", "write", "auth", "slow"
  GET, /api/users,      quick, read
  GET, /api/users/{id},  quick, read
  POST, /api/users,      quick, write
  PUT, /api/users/{id},  quick, write
  DELETE, /api/users/{id}, quick, write
  POST, /api/users/login, quick, auth
```

Następnie:

- `--scope=write` → uruchamia tylko `write` endpoints dla tej usługi
- `--scope=read` → tylko `read`
- `--scope=auth` → tylko `auth`

W `TestQLWatcher` możesz mapować typ pliku na scope:

```python
scope_hint = {
    "orm": "write",
    "schema": "write",
    "route": "read,write",
    "auth": "auth",
}
```

To pozwala **jeszcze bardziej ograniczać zestaw testów** w layer 2, bez touchowania TestQL‑DSL.

## 4. Wizualizacja zależności w browser / terminal

Skoro wypychasz do “browser service”, możesz zrobić:

- **Strona HTML** (lub Web Component + FastAPI proxy) z:
    - live listą usług: `users`, `orders`, `auth`
    - status (✅ OK, ⚠️ Quick fail, ❌ Regressed)
    - link “View last regression” (otwiera `regressions.json` w tabeli)
- **Dashboard typu `--dashboard`**:
    - pokazuje:
        - co ostatnio było zmienione
        - jakie usługi są w `test_queue`
        - jakie testy nie udały się w ostatnich `N` min

Możesz np. użyć prostego `FastAPI` endpointu w `wup`:

```python
@app.get("/dashboard")
def get_dashboard():
    return {
        "services": list(watcher.changed_services),
        "last_regressions": list(regressions[-10:]),
        "queue": [q[^1] for q in watcher.test_queue],
    }
```

i w przeglądarce wgrać prosty Vue/React/Preact front‑end, który to odświeża co 500 ms.

## 5. Opcjonalne, ale bardzo przydatne: offline + fallback

Na Twoim laptopie często chcesz mieć **maksymalną odporność**:

- **`wup --offline`**:
    - wyłącza heartbeat do “browser service”
    - pakuje wszystko do `wup-status.json` i `regressions.json`
    - po podłączeniu internetu synchronizuje (np. jako `wup sync --to browser`)
- **fallback do terminala**:
    - jeśli browser service nie odpowiada, `TestQLWatcher` zapisuje:
        - `wup/current-failure.md` z:
            - `git diff --name-only`
            - `testql --output json` output
            - link do commita
    - `wup ui` → `mdless` lub `bat` w terminalu

***

## 6. Automatyzacja w CI/remote

Kiedy macie już solidną logikę lokalnie, możesz ją upuścić do CI:

- **`wup map-deps`**:
    - generuje `wup-deps.json` → `git add` / `git commit` / `git push`
- **`wup ci-run`**:
    - w CI: `wup ci-run --service users` → wywołuje tylko testy dla tej usługi
    - działa jak `git blame`‑aware, więc nie testuje 200+ endpointów, tylko 3–10
    - jeśli regresja w CI, zapisuje `regressions.json` jako artifact i linkuje w UI

To pozwala uniknąć testowania 200+ endpointów w każdym PR.
